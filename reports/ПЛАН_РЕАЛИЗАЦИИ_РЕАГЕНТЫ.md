### 1.2 Нормы расхода реагентов на тест

```sql
CREATE TABLE analyzer_reagent_rates (
    id VARCHAR(128) PRIMARY KEY,
    analyzer_id VARCHAR(128) NOT NULL REFERENCES analyzers(id),
    reagent_name VARCHAR(255) NOT NULL,       -- "M-52D Diluent"
    operation_type VARCHAR(64) NOT NULL,      -- PATIENT_TEST, STARTUP, SHUTDOWN, QC, CALIBRATION, STANDBY_EXIT_SHORT, STANDBY_EXIT_MEDIUM, STANDBY_EXIT_LONG, BACKGROUND, ENHANCED_CLEAN
    test_mode VARCHAR(64),                    -- "CBC+DIFF", "CBC", null для startup/shutdown
    volume_per_operation_ml DOUBLE PRECISION, -- 27.5 (для жидких)
    units_per_operation INTEGER,              -- 1 (для дискретных: полоски, картриджи)
    unit_type VARCHAR(32) NOT NULL,           -- ML, PIECE, TEST_POSITION
    source_document VARCHAR(512),             -- "Service Manual BC-5150/BC-5000, Table 7-3"
    notes TEXT,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    version BIGINT
);

CREATE INDEX idx_analyzer_reagent_rates_analyzer ON analyzer_reagent_rates(analyzer_id);
CREATE UNIQUE INDEX idx_analyzer_reagent_rates_unique 
    ON analyzer_reagent_rates(analyzer_id, reagent_name, operation_type, COALESCE(test_mode, ''));
```

## Фаза 2: Парсер Applogs

### 2.1 Формат строк (по вашему реальному анализу)

```
Разделитель:    TAB (\t)
Формат времени: Д.ММ.ГГГГ Ч:ММ:СС (без ведущего нуля у часов)
Уровни:         Information: | Debug: | Error: | Warning:
Многострочность: JSON/HL7 может быть на следующей строке без стандартных полей
```

### 2.2 Конечный автомат парсера

```
Состояния:
  IDLE → ожидание нового блока
  SAMPLE_DETECTED → встречен "SAMPLE! {barcode}"
  COLLECTING_RESULTS → сбор JSON результатов (Result object / Save string)
  SAMPLE_COMPLETE → все данные по пробе собраны

Переходы:
  IDLE + "SAMPLE! {barcode}" → SAMPLE_DETECTED (начало блока)
  SAMPLE_DETECTED + "THERE IS NO SR" → помечаем unauthorized
  SAMPLE_DETECTED + "Sample request count: N" → помечаем N направлений
  SAMPLE_DETECTED + "Saving {barcode}" → COLLECTING_RESULTS
  COLLECTING_RESULTS + "Result object to save:" → следующая строка = JSON черновик
  COLLECTING_RESULTS + "Save string:" → следующая строка = JSON финальный (берем отсюда)
  COLLECTING_RESULTS + новый "SAMPLE!" или EOF → SAMPLE_COMPLETE → IDLE
```

## Фаза 3: Сервис классификации проб

### 3.1 Алгоритм SampleClassificationService

```
Вход: ParsedSample (из Applogs или errors.xml)
Выход: Classification + Reason

1. Если hasLisOrder == true И sampleRequestCount > 0:
   → LEGITIMATE, "LIS order found: OrderResearchID={id}"

2. Если hasLisOrder == false И sampleRequestCount == 0:
   a. Если WBC < 0.1 И RBC < 0.1 И HGB < 1:
      → WASH_TEST, "Zero/near-zero results: WBC={wbc} RBC={rbc} HGB={hgb}"
   
   b. Поиск легитимного теста в ±30 мин на том же analyzer_id с тем же test_mode:
      Если найден И результаты WBC/RBC/HGB схожи (±20%):
        → PROBABLE_RERUN, "Correlated with legitimate sample {id} at {time}, Δ={minutes}min"
        → Установить correlated_legitimate_sample_id
   
   c. Если barcode — числа 1-9 или очевидно не штрих-код (<5 символов):
      → SUSPICIOUS, "Non-standard barcode '{barcode}', no LIS order"
   
   d. Иначе:
      → SUSPICIOUS, "No LIS order for barcode '{barcode}'"

3. Если в логе есть "IsOrderProductionControl": true:
   → QC, "Production control flag set"

4. Расчет расхода реагентов по test_mode из analyzer_reagent_rates
```

## Фаза 4: Сервис расчета расхода

### 4.1 ReagentCalculationService

```
Вход: analyzer_id, period_start, period_end
Выход: ReagentConsumptionReport

Шаг 1: Поток 1 (легитимные)
  SELECT classification, test_mode, COUNT(*) 
  FROM parsed_analyzer_samples
  WHERE analyzer_id = ? AND sample_timestamp BETWEEN ? AND ?
    AND classification = 'LEGITIMATE'
  GROUP BY test_mode

  Для каждого test_mode × count:
    Из analyzer_reagent_rates: volume_per_operation × count
    → legitimate_reagent_consumption

Шаг 2: Сервисные расходы (теоретические)
  Рабочих дней в периоде = N
  startup: N × rate(STARTUP)
  shutdown: N × rate(SHUTDOWN)
  background: N × rate(BACKGROUND)
  QC: N × 3 × rate(QC)                    -- 3 уровня
  standby_exit: N × estimated_exits × rate(STANDBY_EXIT_*)
  calibration: estimated_calibrations × rate(CALIBRATION)
  → service_reagent_consumption

Шаг 3: Поток 2 (подозрительные)
  SELECT classification, test_mode, COUNT(*)
  FROM parsed_analyzer_samples
  WHERE analyzer_id = ? AND sample_timestamp BETWEEN ? AND ?
    AND classification IN ('SUSPICIOUS', 'PROBABLE_RERUN', 'WASH_TEST')
  GROUP BY classification, test_mode
  
  Для каждого: count × rate
  → unauthorized_reagent_consumption

Шаг 4: Склад
  inventory_start = остаток на period_start (из reagent_inventory)
  inventory_received = поступления за период
  inventory_end_expected = start + received - legitimate - service - unauthorized
  inventory_end_actual = ввод аналитиком (или USB-данные)
  discrepancy = expected - actual

Шаг 5: Стоимость
  Для каждого реагента: volume × (unit_price / total_volume)
```

- Форма прихода: реагент, лот, количество, дата, цена
- Форма списания: вскрытие, утилизация, порча

### 6.4 Страница: Склад расходных материалов
- Аналогично реагентам, но по категориям (пробирки, наконечники и т.д.)

### 6.5 Страница: Отчет по расходу реагентов (главный дашборд аналитика)
- Выбор анализатора + период
- Три блока:
  1. **Поток 1**: легитимные тесты → расход реагентов → стоимость
  2. **Сервисные**: startup/shutdown/QC/калибровки → расход → стоимость
  3. **Поток 2**: suspicious + rerun + wash → расход → стоимость
- Итого: ожидаемый остаток vs фактический → расхождение
- Таблица подозрительных проб с деталями

### 6.6 Страница: Подозрительные пробы
- Таблица: время, баркод, анализатор, классификация, причина, результаты
- Фильтры: по анализатору, по классификации, по периоду
- Аналитик может вручную переклассифицировать (SUSPICIOUS → RERUN / LEGITIMATE)

## Ключевые принципы реализации

1. **Парсер Applogs** — **конечный автомат**, не regex-поиск. TAB-разделитель, многострочные JSON.
2. **errors.xml** — **StAX/SAX парсинг** (потоковый, не DOM), файлы могут быть >500KB.
3. **Классификация** — детерминированная, с четкими правилами и причинами. Аналитик может переклассифицировать.
4. **Расчет расхода** — по формулам из сервисных мануалов, данные сидируются в БД как справочник.
5. **Сверка** — автоматическая при наличии данных склада, ручной ввод фактического остатка.
6. **Безопасность** — все эндпоинты за авторизацией, операции склада логируются.
7. **Миграция** — Flyway V8, совместима с существующей V1-V7 схемой.
